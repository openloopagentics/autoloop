Only Loops and Sessions exist. A “run” is loops grouped by start-day for display.

Two “self-fixings”

Self-correction (right work) vs self-recovery (staying alive).

01

The vocabulary

How we name the parts.

A Loop is the central noun: one scoped development effort — a single iteration toward the vision. It is a container, not a schedule. An agent drives it; cron does not fire it. Everything else hangs off it.

Terminal cascade

When a loop goes terminal, the server sweeps every non-terminal phase & task to the same status and nulls the current-pointers. No orphaned “running” tasks survive.

endedAt is immutable

Stamped once on the first terminal transition. Never overwritten, even if the loop is later re-touched.

Pointers are derived

currentLoopId / currentPhaseId / currentTaskId recompute as the lowest-order non-terminal child. Never set by hand.

Zombie display rule

A loop running but untouched > 3h renders as paused — so a dead session never reads as live work. Stored status is left untouched.

02

How a loop improves

The self-learning aspects.

A loop doesn’t learn by adjusting hidden weights. It learns by emitting durable, reason-bearing records that the next loop reads. Learning is cross-loop, on the server, and fully inspectable — every record carries a rationale.

The loop acts

Immediately applies the idea, grows the vision, records the defect — and writes down why. It never blocks waiting for permission.

The user vetoes

Accepts, rejects, reorders, or reverts — asynchronously, from the dashboard. The user is never locked out. One pattern: autonomous-with-veto.

Ideas backlog

accept · reject · reorder

At close, the loop proposes ≥ 5 improvement ideas, each with a rationale and origin-loop. The next loop deterministically builds the first accepted, else first proposed — never a rejected one.

reject reverts

03

How a user steers

The sets of guidance — by timescale.

A user guides a loop on three timescales. Across all of them, the user always wins on conflict — a message, a rejection, or a veto overrides the loop’s autonomy.

Vision-time · before the loop

“What good looks like”

Author vision.json via /autoloop-vision: goals, scenarios, rubrics, thresholds, and test commands. The strongest guidance — it defines the target the loop optimizes toward and the bar it must clear.

Mid-run · synchronous steering

The message channel

Send free text (autoloop messages send or the dashboard). The driver drains pending messages at natural boundaries, oldest-first — acting on each, then acknowledging it.

Cross-loop · durable, asynchronous

Steering the trajectory

Between iterations, shape where the product goes without touching a running session: accept / reject / reorder ideas (picks the next loop’s goal), veto vision-change proposals, and review trends & preview to judge whether output is improving.

04

How a loop recovers

“Fixes itself” is two different things.

Keep them apart: self-correction is getting the work right; self-recovery is staying alive.

Self-correction

Getting the work right

quality · within scope
task→ commit→ test-run→ score→ verify→ revise→ bug
↻ re-implement → until the scenario is met, or a revision records why it can’t

A refuted verification or a sub-threshold composite keeps the scenario unmet and feeds the next decision. Honest scoring is a rule — an unmet scenario that triggers a revision is the loop working correctly.

Self-recovery

Staying alive

liveness · durability
  • ResumeA fresh session pulls a state bundle — loop, phases, tasks, scenarios, open bugs, pending messages — and rebuilds its exact position. Step 0, before any setup.
  • RelaunchA SessionEnd hook spawns a fresh headless driver for a non-terminal, non-paused loop — guarded by a project lockfile and crash-loop backoff (≥ 3 relaunches / 30 min stops it).
  • Wake jobA 5-minute job relaunches a paused loop when a user message arrives — so a pause costs no tokens, yet the next message still wakes it.
  • Backstop & zombieThe terminal cascade closes dangling tasks; a stale “running” loop surfaces as paused so a dead session never reads as live.
